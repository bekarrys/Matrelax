# Витрина как POS-рабочее место менеджера

**Дата:** 2026-06-16
**Статус:** согласован, готов к написанию плана

## Контекст и цель

Витрина (бывший клиентский фронтенд) — визуальное «лицо» системы: картинки
товаров, выбор размеров, корзина. Менеджер работает с iPad и собирает заказы
именно через витрину — это удобнее «сухих форм» старой админки. Клиент сам ничего
не оформляет; всё делает менеджер.

**Цель:** превратить витрину в рабочее место менеджера под логином, где сборка
заказа из каталога создаёт управляемый заказ (`adminOrders`), а таблица «История
заказов» остаётся центром просмотра и правок.

## Принятые решения (из ТЗ)

1. Витрину (клиентский фронтенд) сделать доступной **только после авторизации**
   (RoleGuard: `manager`/`admin`).
2. Старый поток создания заказа «сухими формами» (**пошаговый мастер `CreateOrder`,
   `/orders/new`**) и публичный клиентский поток (онлайн-оплата, страница статуса
   покупателя) — **исключить из роутинга и удалить**.
3. Оформление из корзины (`Checkout`) привязать к **`api.adminOrders.create`** по
   POS-логике: выбор точки продаж, **офлайн-оплата** (способ + статус, без
   платёжного шлюза).
4. Таблицу **«История заказов»** (`Orders.jsx` на `adminOrders`) оставить как
   центральное место правок и контроля; правка/разблокировка — существующие.

### Что НЕ входит / не трогаем

Админ-страницы (`/dashboard` аналитика, `/employees`, `/reports`, `/settings`,
`/products` каталог-редактор) — **без изменений**, остаются admin-гейтед. Экран
исполнителя (`/executor`) — без изменений. PWA/чек/печать — отдельно.

## Маршрутизация и доступ

Всё приложение — под логином. `client/src/App.jsx`:

| Путь | Экран | Доступ |
|------|-------|--------|
| `/login` | Login | публичный (форма входа) |
| `/` | **Витрина** (`Home`, каталог) | RoleGuard `manager`/`admin` |
| `/product/:id` | ProductDetail | RoleGuard `manager`/`admin` |
| `/checkout` | **POS-оформление** | RoleGuard `manager`/`admin` |
| `/orders` | **История заказов** (таблица) | RoleGuard `manager`/`admin` |
| `/orders/:id` | OrderDetails (правка/разблокировка) | RoleGuard `manager`/`admin` |
| `/executor` | Executor | `executor` |
| `/dashboard`,`/employees`,`/reports`,`/settings`,`/products*` | админ-страницы | как сейчас (admin) |

- Не вошёл → `/login`. Вошёл `manager`/`admin` → `/` (витрина). `executor` → `/executor`.
- **Удаляются маршруты:** `/orders/new` (мастер), `/order/:id` (статус покупателя),
  `/shop` (витрина переезжает на `/`), `/cart` (был редирект).

## Витрина (каталог) — под логином

`Home` + `ProductDetail` + `CartSheet`/`BottomSheet` остаются как есть визуально,
но рендерятся только после авторизации. Просмотр товаров — из коллекции `products`
(существующий `GET /api/products`). Корзина (Zustand `cartStore`) — без изменений.

Навигация (вместо публичного логотипа): в шапке витрины — переход на **«История
заказов»** (`/orders`); из таблицы — назад на витрину (`/`). Переиспользуем уже
существующий `BrandSwitcher` (переключатель витрина ⇄ заказы), он подходит.

## POS-оформление (переписываем `Checkout.jsx`)

Из корзины «Оформить» → экран оформления, где менеджер заполняет:

- **Точка продаж** — `SALES_POINTS` (Армада/Маденият) — обязательное (новое поле).
- **Имя + телефон** клиента.
- **Способ оплаты** — `PAYMENT_METHODS` (Наличные/Карта/Kaspi/Другое).
- **Статус оплаты** — `PAYMENT_TYPES` (Оплачен/Долг) + сумма оплаты, если «Долг».
- **Получение** — самовывоз/доставка (+адрес при доставке).

На submit → **`api.adminOrders.create(...)`** (authed-клиент `utils/api.js`, НЕ
публичный `api/index.js`). Тело:

```js
{
  salesPoint, customerName, customerPhone,
  items: cart.items.map(i => ({
    name: i.name, modelId: i.productId, size: i.size,
    extra10cm: i.extra10cm, quantity: i.quantity, price: i.price, surcharge: 0,
  })),
  totalAmount,                       // сумма по корзине
  paymentMethod, paymentType,
  paidAmount,                        // = totalAmount при 'paid', иначе введённое
  balance: totalAmount - paidAmount,
  deliveryType, deliveryAddress,
  status: 'new',
}
```
Сервер (`POST /api/admin-orders`) сам проставит `status:'new'`, `itemsUnlocked:false`,
`orderNumber`, `id`, лог `created`. **Онлайн-оплата убирается** (`api.payment.initiate`
не вызывается). После успеха — `clearCart()` → переход на карточку заказа `/orders/:id`.

`canSubmit`: имя, телефон ≥ 10 цифр, выбраны точка продаж и способ оплаты.

## История заказов + правка (оставляем)

`Orders.jsx` (таблица на `adminOrders`) — центр управления: поиск, фильтры, чипы
статусов. Клик по строке → `OrderDetails` (правка свободных полей, блокировка
позиций, разблокировка с причиной, журнал). Всё уже реализовано.

**Мелкая правка `OrderDetails`:** в блоке «Позиции» показывать `item.name` (снимок
из витрины), если он есть; иначе — текущий fallback на catalog-модель. Так заказы
из POS отображают читаемое имя товара.

## Маппинг корзины → заказ

Cart item (`cartStore`): `{ productId, name, size, fabric, extra10cm, price, quantity }`
→ admin item: `{ name, modelId: productId, size, extra10cm, quantity, price, surcharge: 0 }`.
Вынести в чистую функцию `cartToOrderItems(items)` (тестируемо).

## Удаления (по принципу «без спящих файлов»)

Удаляем из роутинга и из репозитория (откат — git/ветка):
- `client/src/pages/CreateOrder/**` — пошаговый мастер.
- `client/src/pages/OrderStatus/**` — статус заказа для покупателя.
- Публичный платёжный путь оформления: вызовы `api.payment.initiate`, `api.orders.create`
  из витрины (заменяются на `adminOrders.create`).
- Маршруты `/orders/new`, `/order/:id`, `/shop`, `/cart` из `App.jsx`.
- Из сайдбара (`Layout`) пункт «Новый заказ» переводится на витрину (`/`) или
  убирается (создание — из витрины).

Сервер: маршруты `routes/orders.js` (`shopOrders`) и `routes/payment.js`
использовались только публичным потоком. В этой итерации **оставляем их
смонтированными как есть** (с фронта не вызываются) — их декоммиссия/удаление это
отдельное решение, вне рамок данной спеки.

## Обработка ошибок / краевые случаи

- Пустая корзина → кнопка оформления недоступна.
- Ошибка `adminOrders.create` → показать сообщение, корзину не чистить.
- Старые `shopOrders` в БД — вне витрины менеджера (не отображаются), не трогаем.

## Тестирование

- **Сервер:** существующие 48 тестов остаются зелёными. Создание заказа уже покрыто
  (`adminOrders.test.js`).
- **Клиент (чистая логика):** добавить `node:test`/юнит на `cartToOrderItems`
  (маппинг корзины → позиции) если вынесем в общий модуль; иначе — ручная проверка.
- **Сборка:** `cd client && npm run build` без ошибок.
- **Сквозной прогон:** логин → витрина → добавить товар → оформить (точка/оплата) →
  заказ появляется в «Истории» со `status:'new'` → правка/разблокировка работает.

## Критерии готовности

- Витрина открывается только после логина (manager/admin); неавторизованный → `/login`.
- Сборка заказа из каталога/корзины создаёт `adminOrder` (POS: точка, офлайн-оплата),
  он виден в «Истории заказов» и редактируется.
- Старый мастер `CreateOrder`, клиентский `OrderStatus`, онлайн-оплата и маршруты
  `/orders/new`,`/order/:id`,`/shop` удалены; меню не ссылается на них.
- Имя позиции из POS отображается в карточке заказа.
- Сборка зелёная, серверные тесты зелёные.
