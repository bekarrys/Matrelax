# Рабочее место менеджера: переосмысление панели заказов

**Дата:** 2026-06-11
**Статус:** согласован, готов к написанию плана

## Контекст и цель

Приложение Matrelax исторически состоит из двух частей: публичного интернет-магазина
(без авторизации) и внутренней панели персонала. Мы переориентируем продукт: основным
рабочим инструментом становится **интерфейс оформления заказа для менеджера**. Менеджер
принимает заказы (по телефону или при встрече), заполняет форму, ведёт их по жизненному
циклу. Админ получает расширенные права (удаление заказов, отчёты, управление).

Публичная витрина выводится из обращения; приложение становится по сути внутренней
CRM/системой управления заказами производства.

### Что НЕ входит в эту итерацию

- Раздел «Склад/Материалы» — отдельный проект позже.
- Физическое удаление серверных маршрутов витрины (`orders.js`, `products.js`,
  `payment.js`) — пока остаются, не вызываются с фронта.
- Физическое удаление файлов страниц витрины на фронте — остаются «спящими» (не
  импортируются/не роутятся) для дешёвого отката.
- Автотесты фронтенда (нет инфраструктуры — вводить не будем).

## Принятые решения

| Тема | Решение |
|------|---------|
| Витрина | Убираем из фронтенд-роутинга; файлы и серверные маршруты не удаляем |
| Правки сервера | Точечные правки `adminOrders.js` разрешены |
| Статусы | Добавляем `new` («Новый»). Поток: `new → progress → ready → delivery → delivered`. Оплата — отдельная колонка |
| Блокировка позиций | Выводится из статуса + временный флаг `itemsUnlocked`; проверка на сервере |
| Разблокировка | Менеджер — с обязательной причиной (в лог); админ — всегда |
| Лог изменений | Детальные полевые диффы `{action:'edit', field, from, to}` |
| Имя клиента | Одно поле `customerName`; поиск по телефону и имени |
| Список заказов | Канбан заменяется таблицей «История заказов» + поиск/фильтры |
| Детали заказа | Остаются отдельной страницей `/orders/:id` |
| Навигация | Сайдбар (существующий каркас), пункты под процесс менеджера |
| Соглашение об именах | camelCase везде (как в текущем коде) |

## Фаза 1 — Данные и бэкенд (фундамент)

### Модель заказа (Firestore `adminOrders`)

| Поле | Сейчас | Станет |
|------|--------|--------|
| `status` | по умолчанию `'progress'` | по умолчанию `'new'` |
| `customerName` | нет | строка, необязательное |
| `itemsUnlocked` | нет | boolean, по умолчанию `false` — разовое разрешение на правку позиций |
| `history[]` | `{ action, by, timestamp }` | расширяется (см. ниже); старые записи остаются валидными |

### Структура записей истории (camelCase)

```js
// Создание заказа
{ at, by, action: 'created' }
// Изменение статуса
{ at, by, action: 'status', from, to }
// Изменение поля заказа (детальный дифф)
{ at, by, action: 'edit', field, from, to }
// Разблокировка позиций
{ at, by, action: 'unlock', reason }
```

`at` — ISO-строка времени, `by` — email пользователя (как в текущем `history`).
Старые записи формата `{ action: 'updated', by, timestamp }` остаются и
отображаются как «Изменён».

### Модель блокировки

Блокировка позиций **выводится из статуса**, отдельного постоянного флага `is_locked`
нет:

- `status === 'new'` → позиции (модель/размер/+10см/количество) редактируются свободно.
- `status !== 'new'` → позиции заблокированы, пока `itemsUnlocked !== true`.
- Прочие поля (телефон, имя, заметки, доставка, скидка, оплата) редактируются всегда.

`itemsUnlocked` — временное разрешение: ставится при разблокировке, сбрасывается в
`false` после сохранения правок позиций.

### Изменения в `server/routes/adminOrders.js`

- **`POST /`** — стартовый статус `'new'`; принимаем и сохраняем `customerName`;
  лог `{action:'created'}`.
- **`PUT /:id`** — сравнить входящие поля со старыми; на каждое изменённое записать
  `{action:'edit', field, from, to}`. **Серверная проверка блокировки:** если в `items`
  есть изменения, а `status !== 'new'` и `itemsUnlocked !== true` → ответ `403`. После
  успешного сохранения с правкой позиций — сбросить `itemsUnlocked` в `false`.
- **`PATCH /:id/unlock`** (новый, роли manager/admin) — body `{reason}`; пустая
  причина → `400`. Ставит `itemsUnlocked: true`, пишет лог `{action:'unlock', reason}`.
- **`PATCH /:id/status`** — добавить переход `new → progress` в `ALLOWED_TRANSITIONS`
  для `manager` и `admin`.

### Серверный воркфлоу

1. **Создание:** `status: 'new'` — заказ «чистый», менеджер вносит все данные.
2. **Передача в работу:** `new → progress` — блокировка позиций включается автоматически.
3. **Запрос правок:** `PATCH /:id/unlock` создаёт лог `unlock`, открывает доступ
   (`itemsUnlocked: true`).
4. **Правка:** `PUT` сверяет старое/новое, пишет диффы в `history`, сбрасывает
   `itemsUnlocked` в `false`.

## Фаза 2 — Таблица «История заказов», поиск и фильтры

Заменяем канбан в `client/src/pages/Orders/Orders.jsx` на таблицу. Логику загрузки
(`api.orders.list()`) сохраняем, меняем только представление.

### Шапка страницы

Заголовок «История заказов» + кнопка «Новый заказ» (→ `/orders/new`). Карточки
статистики (всего / выручка / долги) оставляем.

### Строка управления (над таблицей)

- **Поиск** — один input, клиентская фильтрация по `customerPhone` И `customerName`
  (без учёта регистра, по подстроке) по уже загруженному списку.
- **Фильтр статуса** — чипы: `Все · Новый · В работе · Готов · Доставка · Доставлен`.
- Существующие фильтры по точке продаж и дате — сохраняем.

### Колонки таблицы

| № заказа | Точка продаж | Клиент | Телефон | Тип | Статус | Оплата | Сумма | Дата | → |

- **Точка продаж** — сразу после номера, для быстрого понимания контекста при звонке.
- **Тип** — компактный тег серии/модели первой позиции (напр. «Орто»; при нескольких
  позициях — «Орто +1»).
- **Статус** — `StatusBadge` (добавить стиль для `new`).
- **Оплата** — отдельный бейдж из `PAYMENT_TYPES` (Оплачен/Долг/Аванс), окраска по долгу.
- Клик по строке → `/orders/:id`.

### Поведение

- Сортировка `createdAt desc` сохраняется после поиска и фильтров (фильтрация не меняет
  порядок).
- **Skeleton** строк таблицы на время загрузки.
- Пустой результат → строка «Ничего не найдено».
- Мобильный вид — **горизонтальный скролл** таблицы (не карточки), чтобы видеть 5–7
  заказов списком.

### Константы

`STATUS.NEW = 'new'`, лейбл «Новый», цвета бейджа (`STATUS_LABELS`, `STATUS_COLORS`,
`STATUS_BG`).

## Фаза 3 — Детали заказа, редактирование и блокировка

Дорабатываем существующий `client/src/pages/OrderDetails/OrderDetails.jsx`. Детали
остаются **отдельной страницей** `/orders/:id` (удобно делиться ссылкой, нет проблем с
z-index, комфортно с длинными данными).

### Вычисление состояния (клиент зеркалит сервер)

```js
itemsEditable = order.status === 'new' || order.itemsUnlocked === true
```

### Режим редактирования

- Кнопка **«Редактировать»** доступна всегда.
- Поля телефон/имя/заметки/доставка/скидка/оплата — редактируются всегда.
- Блок **«Позиции»** (модель/размер/+10см/количество):
  - `itemsEditable === true` → поля активны.
  - `itemsEditable === false` → поля задизейблены, сверху бейдж «🔒 Заблокировано
    (в работе)» и кнопка **«Разблокировать для правок»**.
- **«Разблокировать»** → модалка с **обязательным** полем «Причина» →
  `PATCH /:id/unlock {reason}` → перезагрузка заказа (позиции открываются, лог `unlock`).
- **«Сохранить»** → `PUT`; сервер пишет полевые диффы и при правке позиций сбрасывает
  `itemsUnlocked`. После сохранения подтягиваем свежий заказ.

### Карточка «История изменений»

Читаемые строки по типу записи, через словарь меток `FIELD_LABELS`
(`{ size: 'Размер', modelId: 'Модель', customerName: 'Имя клиента', ... }`):

- `created` → «Создан · {by} · {дата}»
- `status` → «Статус: {from} → {to} · {by} · {дата}»
- `edit` → «{поле}: {from} → {to} · {by} · {дата}»
- `unlock` → «🔓 Разблокировано для правок · причина: «{reason}» · {by} · {дата}»
- старые `{action:'updated'}` → «Изменён · {by} · {дата}» (обратная совместимость)

### Права

Менеджер разблокирует с причиной; админ — всегда. Удаление заказа — только админ
(существующий `RoleGuard roles={['admin']}`).

## Фаза 5 — Тестирование и контроль качества

В проекте нет тест-фреймворка. Вводим прагматичный минимум на встроенном `node:test`
(без новых зависимостей), сфокусированный на рисковой логике блокировки и аудита.

### Серверные тесты (`adminOrders.js`, замоканный Firestore `db`)

По TDD — тест пишется до реализации, заодно документирует контракт блокировки:

1. `POST` создаёт заказ со `status: 'new'`, лог `created`.
2. `PUT` правит позиции при `status: 'new'` → успех, в истории `edit`-диффы.
3. `PUT` правит позиции при `status: 'progress'` без `itemsUnlocked` → **403**.
4. `PATCH /:id/unlock` ставит `itemsUnlocked: true` + лог `unlock`; пустая причина → **400**.
5. После `PUT` с правкой позиций `itemsUnlocked` сброшен в `false`.
6. Правка свободных полей (имя/телефон/заметки) при `progress` → успех без разблокировки.
7. `PATCH /:id/status` разрешает `new → progress` для manager/admin.

### Ручной чек-лист фронта

- Поиск по имени и по телефону.
- Чипы фильтра статусов (включая «Новый»).
- Блокировка позиций у заказа в работе.
- Модалка разблокировки с обязательной причиной.
- Читаемый лог изменений (включая старые записи).
- Редиректы: `/` и удалённые витринные пути ведут на `/orders` или `/login`.
- Кнопка «Новый заказ» визуально выделена в сайдбаре.
