# Способ оплаты заказа (paymentMethod)

**Дата:** 2026-06-16
**Статус:** согласован, готов к написанию плана

## Контекст и цель

Менеджер оформляет заказы с iPad. Деньги принимаются **офлайн** — через
физический POS-терминал и Kaspi (QR/перевод). Приложению **не нужно проводить
онлайн-оплату**; его задача — **зафиксировать факт оплаты**: способ + статус.

Статус оплаты уже есть в модели (`paymentType` = Оплачен/Долг/Аванс,
`paidAmount`, `balance`). Не хватает **способа оплаты** (как именно платили).

**Цель:** добавить поле «способ оплаты» в заказ, выбираемое при создании и
редактируемое в любой момент, с записью изменений в журнал.

### Решения, принятые на этапе brainstorming

- QR-картинку в приложении **не храним и не показываем** — она у менеджера офлайн.
- Способ оплаты — **одно поле заказа** (подход A), а не журнал платежей (подход B).
  Покрывает «зафиксировать как платили» и «закрыть долг позже».
- Способы: **Наличные · Карта (терминал) · Kaspi QR/перевод · Другой банк/прочее**.

### Что НЕ входит (YAGNI)

Хранение/показ QR, онлайн-оплата, Kaspi Bisnes API, чек/печать/share, изменения
PWA, журнал частичных платежей (`payments[]` — возможный апгрейд B позже).

---

## Модель данных

Заказ (`adminOrders`) получает поле:

| Поле | Тип | Значения |
|------|-----|----------|
| `paymentMethod` | string, опциональное | `cash` \| `card` \| `kaspi` \| `other` |

- Старые заказы без поля → в UI показываем «—».
- Неизвестное значение → показываем как есть (raw).
- **Свободное поле:** редактируется всегда, **не** подпадает под блокировку
  позиций (блокируются только модель/размер/+10см/количество).

## Изменения по файлам

### 1. Константы — `client/src/utils/constants.js`

```js
export const PAYMENT_METHODS = {
  cash:  'Наличные',
  card:  'Карта (терминал)',
  kaspi: 'Kaspi QR / перевод',
  other: 'Другой банк / прочее',
};
```
В `FIELD_LABELS` добавить: `paymentMethod: 'Способ оплаты'`.

### 2. Аудит — `server/utils/orderLogic.js`

В массив `TRACKED_FIELDS` добавить `'paymentMethod'` и `'paymentType'`. Тогда
смена способа/статуса оплаты при `PUT` пишется в историю как
`{action:'edit', field, from, to}`. Существующие транзакции и журнал не меняются.

### 3. Создание — `client/src/pages/CreateOrder/CreateOrder.jsx`

В шаге «Оплата» под выбором `paymentType` (Оплачен/Долг/Аванс) добавить выбор
способа: плитки в стиле `.point-buttons` (крупные, удобно на iPad), значения из
`PAYMENT_METHODS`. Выбор сохраняется в `form.paymentMethod` и уходит в
`api.adminOrders.create`. Дефолт — `cash`.

### 4. Карточка — `client/src/pages/OrderDetails/OrderDetails.jsx`

- **Просмотр:** в блоке «Получение»/«Финансы» строка
  `Способ оплаты: {PAYMENT_METHODS[order.paymentMethod] || '—'}`.
- **Правка:** в `editMode` — селект/плитки способа (свободное поле). Сохранение
  идёт через существующий `PUT` → сервер пишет дифф в журнал. Закрывает кейс
  «Долг → доплатил → Оплачено · Карта».

### 5. Список заказов — без изменений

Бейдж оплаты (`paymentInfo`) остаётся как есть; способ виден в карточке заказа.

## Поток данных

```
CreateOrder (выбор paymentMethod) → POST /admin-orders → сохранён в заказе
OrderDetails (правка paymentMethod) → PUT /admin-orders/:id
  → runTransaction (атомарно)
  → computeFieldDiffs: paymentMethod изменён → {action:'edit', field, from, to} в history
OrderDetails «История изменений» → показывает «Способ оплаты: Наличные → Карта»
```

## Обработка ошибок / краевые случаи

- `paymentMethod` отсутствует → «—», не ошибка.
- Значение вне словаря → показываем raw-строку.
- Редактирование способа у заказа «в работе» — **разрешено** (свободное поле),
  блокировка позиций на него не распространяется.

## Тестирование

- **Сервер:** расширить `server/test/orderLogic.test.js` — `computeFieldDiffs`
  фиксирует изменение `paymentMethod` (from/to). `npm test` (в `server/`) зелёный.
- **Клиент:** `npm run build` без ошибок; ручная проверка: выбор способа при
  создании, отображение и правка в карточке, запись в журнал.

## Критерии готовности

- В шаге «Оплата» можно выбрать один из 4 способов; он сохраняется в заказе.
- В карточке заказа способ виден и редактируется; правка пишется в журнал
  читаемой строкой через `FIELD_LABELS`.
- Старые заказы без способа отображаются как «—», ничего не падает.
- `paymentMethod`/`paymentType` отслеживаются в аудите; тесты зелёные.
