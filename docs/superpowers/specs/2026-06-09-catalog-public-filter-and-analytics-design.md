# Каталог: публичный фильтр по сериям + аналитика по товарам

**Дата:** 2026-06-09
**Статус:** Phase 1 — к реализации; Phase 2 — направление зафиксировано, детальный брейншторм отдельным циклом.

## Контекст

Админка каталога завершена: товары имеют `series` и `isActive`, бэкенд `GET /api/products`
отдаёт записи отсортированными (серия → имя) и поддерживает `?activeOnly=1`. Публичная витрина
([client/src/pages/Home/Home.jsx](../../../client/src/pages/Home/Home.jsx)) грузит товары через
публичный модуль `client/src/api/index.js` (`api.products.list(category)` уже с `activeOnly=1`)
и показывает их рядами внутри вкладок-категорий (матрасы / топперы / наматрасники / подушки).

Два вектора развития: (1) дать покупателю фильтр по сериям на витрине; (2) дать админу аналитику
«что лучше продаётся в серии X». Решено: специфицировать оба, строить по очереди — сперва Phase 1.

## Phase 1 — Фильтр по сериям на витрине (к реализации)

### Цель
Покупатель внутри выбранной категории может отфильтровать товары по серии через компактный
дропдаун «Серия ▾» рядом с вкладками категорий.

### Ключевое решение: фильтрация на клиенте
Home уже загружает все товары категории; `series` присутствует в каждом товаре; набор на категорию
мал (≤ ~12, реалистично до 100). Фильтр — это `products.filter(p => p.series === selected)`
в памяти: мгновенно, без сети, без новых эндпоинтов.

**Отвергнуто:** бэкенд-параметр `?series=` — лишние сетевые round-trip'ы и задержка ради нулевой
выгоды. Эндпоинт остаётся неизменным.

### Компоненты
- **`client/src/components/SeriesDropdown/SeriesDropdown.jsx`** — презентационный дропдаун.
  - Вход (props): `series` (string[]), `value` (string, `'all'` = «Все»), `onChange(value)`.
  - Поведение: кнопка «Серия ▾» / «<выбранная серия> ▾»; раскрывает список «Все» + серии;
    закрывается по выбору и по клику вне (обработчик на document).
  - Зависимости: только React + lucide-react (как остальной UI). Без знания о товарах.
- **`Home.jsx`** — владелец состояния и логики фильтрации.

### Поток данных
1. Home получает товары `activeCategory` (существующая логика, без изменений).
2. Вычисляет `seriesList` = уникальные `series` из выдачи, отсортированные (`localeCompare 'ru'`).
3. Состояние `selectedSeries`, дефолт `'all'`.
4. При смене категории → `selectedSeries` сбрасывается в `'all'` (серии в категориях разные).
5. Отображаемые товары: `selectedSeries === 'all' ? products : products.filter(p => p.series === selectedSeries)`.

### Краевые случаи
- **0–1 уникальных серий** в категории → дропдаун **скрыт** (фильтровать нечего).
- **Идёт загрузка** → дропдаун скрыт (рендерится после прихода данных).
- **Пустой результат фильтра** → существующая разметка пустого/скелетон-состояния Home.
- Товары без `series` (теоретически) → попадают в группу при derive; на витрине их не должно быть,
  т.к. бэкенд проставляет дефолт `series` при создании.

### Что НЕ входит в Phase 1 (YAGNI)
- Сортировка (по цене и т.п.) — отдельная фича при необходимости.
- Мультивыбор серий, фильтры по цене/жёсткости.
- Изменения бэкенда.

### Верификация
В проекте нет тест-раннера. Проверка ручная + Playwright (как с админ-каталогом):
выбрать категорию → выбрать серию → список фильтруется; сменить категорию → фильтр сбросился
в «Все»; категория с одной серией → дропдаун скрыт; 0 ошибок в консоли.

## Phase 2 — Аналитика по товарам (направление; детальный брейншторм — отдельный цикл)

### Цель
Админ отвечает на вопрос «какие модели лучше всего продаются (в серии X)»: топ моделей по выручке
и по штукам за период.

### Источник данных
`adminOrders.items[]` (поля `modelId`, `quantity`, `price`) сшиваются с `products` по
`modelId → series`. Агрегация на бэкенде, новый эндпоинт-агрегатор
(напр. `GET /api/reports/products?period=`), права `admin` (или `admin+manager`, уточнить).
Фронт — таблица/виджет в AdminDashboard или новой вкладке отчётов.

### Главный риск (подтверждён, прорабатываем в цикле Phase 2)
**Консистентность `modelId` между заказами и каталогом.** Если матрас удалён или пересоздан с новым
ID, статистика по старым заказам «поедет» (потеря связи модель→серия, осиротевшие `modelId`).
Варианты решения к обсуждению:
- неизменяемый `productSnapshot` (name, series, modelId, price) внутри позиции заказа на момент покупки;
- историческая/мягкая связь (soft-reference) вместо жёсткого `modelId`;
- запрет жёсткого удаления товаров с историей заказов (только `isActive=false`).

### Открытые вопросы для цикла Phase 2
- Какие именно метрики и срезы (выручка vs штуки; период; по сериям и/или по точкам продаж).
- Где живёт UI (вкладка в AdminDashboard vs отдельная страница).
- Права доступа к товарной аналитике.
